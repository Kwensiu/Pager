# Sidebar 存储迁移方案（MVP）

## 适用前提

当前项目暂无真实用户使用，目标是避免过度设计，先保证：

1. 升级不容易把现有数据搞坏。
2. 出错时可快速恢复。
3. 后续可平滑升级到完整方案。

## 本期保留（必须做）

1. `storeSchemaVersion`
- 初始化写入 `1`。
- 后续结构变化按 `vN -> vN+1` 追加。
- 本期不支持降级运行：检测到旧版本读取更高 `storeSchemaVersion` 时，直接退出（不允许写入）。

2. 一次性桥接（localStorage -> electron-store）
- 渲染层启动时上报历史键。
- 冲突规则固定：`electron-store` 优先，仅在主存储缺值时回填 bridge 值。

3. 最小迁移执行器
- 启动顺序：先 bridge，再 migrate。
- 主进程等待 bridge 最长 3 秒；超时按 bridge 失败处理并继续 migrate。
- 迁移函数保持幂等。

4. 最小备份与回滚
- 每次迁移前做 1 份全量快照。
- 快照最小键集合：`primaryGroups`、`settings`、`windowState`、`lastActiveWebsiteId`、`shortcuts`、`storeSchemaVersion`。
- 迁移失败时回滚该快照并提示用户。

## 本期暂缓（不做）

1. staged 事务提交 / commit marker。
2. 复杂超时编排与观测体系。
3. 只读模式与复杂降级引导流程。
4. 多层结构断言自动修复链路。

## 失败处理（简化）

1. bridge 失败：记录日志，继续迁移。
2. migrate 失败：自动回滚最近快照，提示“迁移失败，已恢复”。
3. 回滚失败：提示“请重置数据”，并保留错误日志。

## 最小测试清单

1. 旧数据存在时，bridge 后不覆盖已存在的 store 值。
2. 迁移函数重复执行结果一致（幂等）。
3. 迁移失败后可回滚到迁移前状态。

## 未来升级触发条件

当满足任一条件时，从 MVP 升级到完整方案：

1. 准备公开发布或已有真实用户数据。
2. 出现多次迁移失败或数据恢复事件。
3. 需要支持严格审计与 crash-safe 原子提交。

升级方向：

1. 引入 staged 事务提交。
2. 完整快照保留策略与自动清理。
3. 只读保护、降级策略、可观测指标。
